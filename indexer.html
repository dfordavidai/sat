⟶ How to get Supabase credentials
1
Go to supabase.com → sign in → click New Project
SQL
Run this SQL in Supabase SQL Editor to create required tables:
-- LinkCore v13 required tables
create table if not exists lc_projects (
  id text primary key,
  name text,
  note text,
  created_at timestamptz default now()
);
create table if not exists lc_links (
  id text primary key,
  project_id text references lc_projects(id) on delete cascade,
  original text,
  code text,
  verdict text default 'pending',
  hits integer default 0,
  submitted boolean default false,
  retry_count integer default 0,
  added_at timestamptz default now(),
  tag text
);
create table if not exists ic_short_links (
  code text primary key,
  target text,
  hits integer default 0
);
-- Config store (settings, SA vault, channel toggles)
create table if not exists lc_config (
  key text primary key,
  value text,
  updated_at timestamptz default now()
);
-- Per-domain sitemaps
create table if not exists lc_sitemaps (
  domain text primary key,
  sitemap_url text,
  xml text,
  url_count integer default 0,
  updated_at timestamptz default now()
);
-- Enable RLS (disable for anon key usage)
alter table lc_projects enable row level security;
alter table lc_links enable row level security;
alter table ic_short_links enable row level security;
alter table lc_config enable row level security;
alter table lc_sitemaps enable row level security;
create policy "anon all" on lc_projects for all using (true) with check (true);
create policy "anon all" on lc_links for all using (true) with check (true);
create policy "anon all" on ic_short_links for all using (true) with check (true);
create policy "anon all" on lc_config for all using (true) with check (true);
create policy "anon all" on lc_sitemaps for all using (true) with check (true);
2
Set a name, database password and region → wait for it to provision (~1 min)
3
Go to Project Settings → API → copy the Project URL and anon public key
4
Run the setup SQL in the SQL Editor tab — see the README for the full block

⟶ How to set up redirect domains on Vercel
1
Create a GitHub repo for each domain with two files:

vercel.json
{"routes":[{"src":"/link/(?<code>[^/]+)","dest":"/api/redirect?code=$code"}]}
api/redirect.js — the Edge Function (download from LinkCore setup kit or copy from the instructions below)
2
Go to vercel.com → Add New → Project → import the repo → before deploying, go to Settings → Environment Variables and add:

SUPABASE_URL = your Supabase project URL
SUPABASE_KEY = your Supabase anon key

Then deploy.
3
In the Vercel project go to Settings → Domains → add your custom domain → copy the DNS records to your registrar
4
Repeat for all 4 domains. The api/redirect.js file is identical for all — only the env vars differ if needed. Enter the full base URLs above including /link/ at the end.
✓
Why this works: The Edge Function returns HTTP 301 Location: <original-url>. Googlebot follows 301s server-side, crawls and indexes the destination page — not your short URL. JS-based redirects (the old method) don't reliably pass this signal.

⟶ How to get a GSC Service Account key
1
Go to console.cloud.google.com → create or select a project
2
Go to APIs & Services → Enable APIs → search for Google Search Console API → enable it
3
Go to IAM & Admin → Service Accounts → click + Create Service Account → give it a name → click Done
4
Click the service account → go to Keys tab → Add Key → Create new key → JSON → a JSON file downloads
5
Go to search.google.com/search-console → open your property → Settings → Users and permissions → Add user → paste the client_email from the JSON → set permission to Owner
6
Paste the entire contents of the downloaded JSON file into the SA Key box above and click + Add Key
